> Since you’re in Cross Platform App Development, understanding event loops is foundational — especially for frameworks like Node.js, React Native, Flutter, and even browser-based apps.
>
> Let’s break it down clearly and practically.
>
> 🔁 **What Is an Event Loop?**  
> An event loop is a programming pattern that allows a single thread to handle many tasks efficiently using asynchronous operations.
>
> Instead of blocking (waiting) for tasks like:  
> - Network requests  
> - File reads  
> - Timers  
> - User input  
> - Animations  
> …it registers them and continues running. When those tasks finish, they are placed in a queue and executed later.
>
> 🧠 **Why Do We Need It?**  
> JavaScript (and Dart in Flutter) runs mostly on a single thread.
>
> If it didn’t use an event loop:  
> ```javascript
> fetchData(); // waits 3 seconds
> print("Hello");
> ```  
> The UI would freeze for 3 seconds. The event loop prevents that freeze.
>
> 🏗 **Core Components of the Event Loop**  
> Think of it like this:  
> Call Stack → Web APIs / Native APIs → Callback Queue → Event Loop
>
> 1️⃣ **Call Stack** — Where synchronous code runs.  
> ```javascript
> console.log("Hi");
> ```  
> Runs immediately on the stack.
>
> 2️⃣ **Web APIs / Native APIs** — Handles async tasks like setTimeout, HTTP requests, File system, Gesture input.
>
> 3️⃣ **Callback Queue** — When async work finishes, its callback goes here.
>
> 4️⃣ **Event Loop** — Continuously checks: "Is the call stack empty?"  
> If yes → it pushes the next callback from the queue into the stack.
>
> 📦 **JavaScript Example (Important for React Native)**  
> ```javascript
> console.log("Start");
>
> setTimeout(() => {
>     console.log("Inside timeout");
> }, 0);
>
> console.log("End");
> ```  
> Output:  
> Start  
> End  
> Inside timeout  
> Even though timeout is 0, it’s still async — it waits until the stack is empty.
>
> 🧵 **Microtasks vs Macrotasks (Advanced but Important)**  
> In environments like Node.js and browsers:  
> **Macrotasks:** setTimeout, setInterval, setImmediate, I/O  
> **Microtasks:** Promises (.then, async/await), queueMicrotask  
> Microtasks run before macrotasks.
>
> Example:  
> ```javascript
> setTimeout(() => console.log("Timeout"), 0);
> Promise.resolve().then(() => console.log("Promise"));
>
> console.log("End");
> ```  
> Output:  
> End  
> Promise  
> Timeout  
> Microtasks have higher priority.
>
> 📱 **Event Loop in Cross-Platform Frameworks**  
> ⚛ React Native  
> Uses JavaScript event loop + a bridge to native code.  
> If you block the JS thread → UI freezes.  
> Bad example: `while(true) {}` → App freezes completely.
>
> 🐦 Flutter (Dart Event Loop)  
> Flutter uses Event queue + Microtask queue. Dart’s event loop works similarly to JavaScript’s.  
> ```dart
> Future(() => print("Event"));
> scheduleMicrotask(() => print("Microtask"));

> ```  
> Output: End, Microtask, Event

> 🎯 **Real-World Example (Mobile App)**  
> User taps a button:  
> Tap event goes into event queue → Event loop picks it → API request starts (async) → UI stays responsive → When API finishes → callback updates UI  
> Without event loop → app would freeze during the API call.
>
> 🚨 **What Blocks the Event Loop?**  
> - Heavy loops  
> - Large JSON parsing  
> - Big synchronous computations  
> - Image processing on main thread  
> In React Native → Use InteractionManager or move work native-side  
> In Flutter → Use isolates
>

> Doesn’t stand waiting in kitchen

> Waiter delivers food

> Event Loops Explained  
> An event loop is the mechanism that allows programs to handle multiple things happening over time — like user taps, network responses, or timers — without freezing up.

>
> **The Core Idea**  
> Your app runs on a single main thread. Without an event loop, any slow operation (like fetching data from an API) would freeze the entire UI. The event loop solves this by saying:

> - Call Stack — where your code currently runs, one thing at a time  
> - Web APIs / Platform APIs — handle async work off the main thread (timers, HTTP calls, I/O)  
> - Queue — completed async tasks line up here, waiting their turn  
> - Event Loop — constantly checks: "Is the call stack empty? If yes, pull the next task from the queue."
>
> Here's a simple example in Dart (Flutter):
> ```dart
> print("1. App starts");
>
> Future.delayed(Duration(seconds: 2), () {
>     print("3. Network response arrived");
> });
>
> print("2. UI is still responsive!");
> ```
>
> // Output:  
> // 1. App starts  
> // 2. UI is still responsive!  
> // 3. Network response arrived  ← arrives after 2 seconds  
> The UI never froze — the event loop handled the wait in the background.
>
> **Why It Matters for Cross-Platform Dev**  
> FrameworkEvent Loop ImplementationFlutter/DartDart's event loop + microtask queueReact NativeJavaScript event loop (same as browsers)Kotlin MultiplatformCoroutines (structured concurrency on top of event loops)  
> In Flutter, there are actually two queues:  
> - Microtask queue — high priority (e.g., Future.microtask)  
> - Event queue — normal async tasks (timers, I/O, user input)  
>
> Microtasks always drain before the next event is processed.
>
> **Common Pitfalls to Avoid**  
> - Blocking the main thread — never run heavy computation synchronously (use Isolates in Flutter or Worker threads in RN)  
> - Callback hell — deeply nested callbacks; solved by async/await

> ```dart

>     final user = await fetchFromAPI(); // event loop handles the wait
>     print(user.name);                  // runs when stack is free
> }
> ```
>
> The event loop is the heartbeat of every reactive UI framework you'll use. Once you internalize it, concepts like async/await, streams, and state management patterns all click into place.

> Closures are one of the most important (and misunderstood) concepts in JavaScript. Once you understand them, patterns like modules, callbacks, and React hooks become much clearer.
>
> **1. What Is a Closure?**
>
> A closure is created when a function “remembers” variables from its lexical scope even after the outer function has finished executing.
>
> In simple terms:
>
> A closure lets a function access variables from an outer function after that outer function has returned.
>
> **2. Basic Example**
>
> ```javascript
> function outer() {
>   let count = 0;
>
>   return function inner() {
>     count++;
>     console.log(count);
>   };
> }
>
> const counter = outer();
>
> counter(); // 1
> counter(); // 2
> counter(); // 3
> ```
>
> Why does this work?
>
> - outer() runs and creates count  
> - It returns inner  
> - Normally, count would be destroyed when outer() finishes  
> - But because inner references count, JavaScript keeps it alive  
>
> That preserved scope is the closure.
>
> Each call to outer() creates a new closure:
>
> ```javascript
> const counter1 = outer();
> const counter2 = outer();
>
> counter1(); // 1
> counter1(); // 2
>
> counter2(); // 1 (separate state)
> ```
>
> **3. How Closures Actually Work**
>
> Closures are based on lexical scoping.
>
> JavaScript determines variable access based on where functions are defined, not where they are called.
>
> ```javascript
> let globalVar = "I'm global";
>
> function outer() {
>   let outerVar = "I'm outer";
>
>   function inner() {
>     console.log(globalVar);
>     console.log(outerVar);
>   }
>
>   return inner;
> }
>
> const fn = outer();
> fn();
> ```
>
> Even though outer() has finished, inner() still has access to:
>
> - outerVar  
> - globalVar  
>
> Because it was defined inside outer.
>
> **4. Practical Examples**
>

>
> Closures are often used to simulate private variables.
>

>       balance -= amount;
>     },
>     getBalance() {
>       return balance;
>     }
>   };
> }
>
> const account = createBankAccount(100);
>
> account.deposit(50);
> console.log(account.getBalance()); // 150
> ```
>
> Here:
>
> - balance cannot be accessed directly  
> - It’s only accessible through the returned methods  
>
> This is data privacy via closure.
>
> **Example 2: Function Factories**
>
> ```javascript
> function multiplyBy(factor) {
>   return function(number) {
>     return number * factor;

>

> const triple = multiplyBy(3);
>
> console.log(double(5)); // 10
> console.log(triple(5)); // 15

>

>

> ```

>

> 3  
> 3

> Even after delayedGreeting finishes, name is remembered.

>

>
> - The function code  
> - A reference to the environment where it was created  
>
> That environment is the closure.
>

>
> A function + its backpack of remembered variables.
>

> A closure happens when:
>
> - A function is defined inside another function  
> - The inner function references variables from the outer function  
> - The inner function survives after the outer function returns  
>
> Closures enable:
>
> - Data privacy  
> - Function factories  
> - Stateful functions  
> - Async callbacks  
>
> Common issues:
>
> - var in loops

The following explanation was generated by Claude without modifications:

> **Closures in JavaScript**
>
> A closure is a function that "remembers" the variables from its outer (enclosing) scope even after that outer function has finished executing. This happens because functions in JavaScript carry a reference to their surrounding lexical environment.
>
> **How Closures Work**
>
> When a function is defined, it captures a reference to its outer scope — not a copy of the values, but a live link to the variables themselves.
>
> ```javascript
> function makeCounter() {
>   let count = 0; // This variable is "closed over"
>
>   return function () {
>     count++;
>     return count;
>   };
> }
>
> const counter = makeCounter();
> counter(); // 1
> counter(); // 2
> counter(); // 3
> ```
>
> makeCounter has finished running, yet count persists because the returned function holds a closure over it.
>
> **Practical Examples**
>
> **1. Data privacy / encapsulation**
>
> Closures let you create private state without classes:
>
> ```javascript
> function createBankAccount(initialBalance) {
>   let balance = initialBalance; // private — not accessible from outside
>
>   return {
>     deposit(amount)  { balance += amount; },
>     withdraw(amount) { balance -= amount; },
>     getBalance()     { return balance; },
>   };
> }
>
> const account = createBankAccount(100);
> account.deposit(50);
> account.getBalance(); // 150
> account.balance;      // undefined — truly private
> ```
>
> **2. Function factories**
>
> Generate specialized functions from a template:
>
> ```javascript
> function multiplier(factor) {
>   return (number) => number * factor;
> }
>
> const double = multiplier(2);
> const triple = multiplier(3);
>
> double(5); // 10
> triple(5); // 15
> ```
>
> **3. Memoization / caching**
>
> Cache expensive results between calls:
>
> ```javascript
> function memoize(fn) {
>   const cache = {};
>   return function (n) {
>     if (cache[n] !== undefined) return cache[n];
>     cache[n] = fn(n);
>     return cache[n];
>   };
> }
>
> const expensiveCalc = memoize((n) => n * n);
> expensiveCalc(10); // computed: 100
> expensiveCalc(10); // cached:   100
> ```
>
> **4. Partial application**
>
> Pre-fill some arguments of a function:
>
> ```javascript
> function add(a, b) { return a + b; }
>
> function partial(fn, ...presetArgs) {
>   return (...laterArgs) => fn(...presetArgs, ...laterArgs);
> }
>
> const add10 = partial(add, 10);
> add10(5);  // 15
> add10(20); // 30
> ```
>
> **Common Pitfalls**
>
> **1. The classic loop bug (with var)**
>
> This is the most infamous closure trap. var is function-scoped, so every iteration shares the same i:
>
> ```javascript
> // ❌ Broken — all callbacks print 3
> for (var i = 0; i < 3; i++) {
>   setTimeout(() => console.log(i), 100);
> }
>
> // ✅ Fixed with let — block-scoped, each iteration gets its own i
> for (let i = 0; i < 3; i++) {
>   setTimeout(() => console.log(i), 100); // 0, 1, 2
> }
>
> // ✅ Also fixed with an IIFE (pre-ES6 approach)
> for (var i = 0; i < 3; i++) {
>   ((j) => setTimeout(() => console.log(j), 100))(i);
> }
> ```
>
> **2. Accidentally sharing mutable state**
>
> All closures over the same variable share mutations — this can be surprising:
>
> ```javascript
> function makeAdders() {
>   let x = 0;
>   const add1 = () => ++x;
>   const add2 = () => ++x; // same x!
>   return { add1, add2 };
> }
>
> const { add1, add2 } = makeAdders();
> add1(); // 1
> add2(); // 2  ← mutated the shared x
> add1(); // 3  ← not 2!
> ```
>
> **3. Memory leaks**
>
> Closures keep their entire outer scope alive. If a closure is long-lived, it prevents garbage collection of everything it references:
>
> ```javascript
> function attachHandler() {
>   const hugeArray = new Array(1_000_000).fill("data");
>
>   document.getElementById("btn").addEventListener("click", () => {
>     console.log(hugeArray[0]); // hugeArray stays in memory as long as the listener lives
>   });
> }
> ```
>
> Fix by nulling references you no longer need, or removing event listeners when done.
>
> **4. Stale closures in React hooks**
>
> In React, effects/callbacks can capture an old value of state or props:
>
> ```javascript
> // ❌ count is always 0 inside the interval
> useEffect(() => {
>   const id = setInterval(() => {
>     console.log(count); // stale!
>   }, 1000);
>   return () => clearInterval(id);
> }, []); // empty deps — closure captures initial count
>
> // ✅ Include count in deps, or use the functional updater form
> useEffect(() => {
>   const id = setInterval(() => {
>     setCount(c => c + 1); // updater form avoids capturing stale state
>   }, 1000);
>   return () => clearInterval(id);
> }, []);
> ```
>
> **Mental Model Summary**
>
> Concept | What it means  
> Lexical scope | A function can see variables where it was defined, not where it's called  
> Closure | A function + the environment it was defined in  
> Shared reference | Multiple closures over the same variable share mutations  
> Lifetime | A closed-over variable lives as long as any closure referencing it does  
>
> The key insight: closures capture variables by reference, not by value. Once that clicks, most closure bugs become immediately obvious.

> The DOM event model defines how events (like clicks, key presses, or scrolls) travel through the document and how JavaScript can respond to them.
>
> Understanding this is essential for writing clean, predictable frontend code.
>
> **1️⃣ The Three Phases of the DOM Event Flow**
>
> When an event happens (e.g., clicking a button), it doesn’t just trigger on that element. The event travels through the DOM in three phases:
>
> Window → Document → <html> → <body> → ... → Target → ... → <body> → <html> → Document → Window
>
> **Phase 1: Capturing Phase (Trickling Down)**
>
> - The event starts at the window  
> - Travels down the DOM tree  
> - Stops when it reaches the target element  
>
> 👉 Rarely used intentionally, but powerful when needed.
>
> **Phase 2: Target Phase**
>
> The event reaches the actual element that triggered it.
>
> Event listeners on the target execute.
>
> **Phase 3: Bubbling Phase (Most Common)**
>
> - The event bubbles upward from the target  
> - Travels back up through parent elements  
> - Ends at window  
>
> 👉 This is the default and most commonly used phase.
>
> **2️⃣ Example**
>
> ```html
> <div id="parent">
>   <button id="child">Click me</button>
> </div>
> ```
>
> ```javascript
> document.getElementById("parent").addEventListener("click", () => {
>   console.log("Parent clicked");
> });
>
> document.getElementById("child").addEventListener("click", () => {
>   console.log("Child clicked");
> });
> ```
>
> When clicking the button:
>
> Console output:
>
> Child clicked  
> Parent clicked
>
> Why?
>
> - Target phase → child runs  
> - Bubbling phase → parent runs  
>
> **3️⃣ Enabling Capturing**
>
> By default, listeners run in the bubbling phase.
>
> To listen during capturing:
>
> ```javascript

> ```
>
> Or modern syntax:
>
> ```javascript

> ```
>

>

>
> Sometimes you don’t want the event to continue bubbling.
>

>
> ```javascript

> ```
>
> Stop everything (including other handlers on same element):
>
> ```javascript

> ```

>   }
> });
> ```
>
> Why this works:
>
> Because of bubbling.
>
> Benefits:
>
> - Better performance  
> - Works for dynamically added elements  
> - Cleaner code  
>
> 👉 Used heavily in:
>
> - Vanilla JS apps  
> - React (synthetic events)  
> - Vue.js  
> - Angular  
>

>

>

> ```
>
> Here:
>
> - Click inside modal → stop bubbling

>
> ```javascript

> });
> ```
>
> This stops:

> - Navigation

>
> Common for:

>|Start|`^`|Beginning of string|

>|End|`$`|End of string|

>A few important notes about this approach:
>
>**What it covers well**: The regex handles the vast majority of real-world email formats — standard addresses, subdomains (`user@mail.google.com`), dots/hyphens/underscores in the local part, and multi-part TLDs (`.co.uk`).
